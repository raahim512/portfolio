Classboards:

README.md – ▢ ⨉

What is it?:

A forum/messageboard platform designed to bridge communication within the HHS Community.

(Please excuse the horrendous UI, I put it together in around a week since I was running out of time and spent most of it learning Flask and SQLite. Plus, this is 2 years old.)

How does it work?

Classboards is built in Flask and uses the SQLite database system. With a frontend created in pure HTML and CSS and zero JS.

It uses Flask's Jinja templating engine to pass data between the backend (Python) and the frontend. Each page is dynamically rendered using templates, with content such as posts and account info pulled directly from the SQLite database.

This database stores information such as user accounts (usernames, emails, passwords, age, and teacher/parent status) and posts (titles, content, and associated usernames).

Now, why exactly would it need to know the Parent/Teacher status? That's because there's a dedication section for posts known as the Parent Bridge.

PyCorrupt:

README.md – ▢ ⨉

What is it?:

A small but effective wiperware malware prototype written in Python.

(Just a reminder: I created this purely for educational purposes to learn about the Windows API and other low-level system behavior, I will not be sharing the destructive functionality, but only how the visuals work.)

How does it work?

Basically, Windows uses something called GDI (Graphics Device Interface) to handle all its drawing operations, from rendering windows and buttons to drawing directly on the screen. With Python's pywin32 library, you can tap into GDI functions like PatBlt, BitBlt and many more to create intense visual effects that look like they're straight out of a stereotypical computer-exploding virus. This project uses loops and screen resolution-based calculations to spam the display with boxes, distortions and flashing shapes at high speeds.

The way the destructive part works is by overwriting the Master Boot Record (MBR), which is the first 512 bytes of the physical drive where Windows is installed, with custom hex (basically garbage random) data. This MBR contains the partition table and bootloader, without it, your system has no idea where to locate or load the operating system, effectively making it unbootable and barely repairable.

How dangerous is this? Is my morality put into question for it?

For the latter, that's up to you.

For how dangerous it actually is. It's quite dangerous. Let me explain:

The way the destructive code works is by overwriting the first 512 bytes of the raw physical disk (Not just a partition like C:/, but the entire drive).

This is what the MBR is structured like:

Section Size (bytes) Description
Bootloader Code 446 First-stage boot code
Partition table 64 Four 16-byte entries
Boot signature 2 Always 0x55AA

My code wipes all 512 bytes. The entire partition layout is lost and thus makes it unreadable by most OSes.

The light at the end of the road:

Adding a little PSA here, if you ever find yourself to fall victim to such a horrible fate. You still have the choice of using recovery tools such as "TestDisk" or "DMDE" to recover your data. Good luck though!
